# JIRA Ticket Format Reference — For Spec Updater

When the spec-updater generates JIRA tickets alongside change proposals, it uses the **same 6-section ticket anatomy** as the jira-generator skill.

## Ticket Structure

Every ticket generated by the spec-updater must use this exact format:

**Summary:** `[Action Verb] + [Specific Feature/Function] + [Target Area]`

### 📋 What Currently Exists
Describe the current system state from the functional spec. Reference specific files and IDs.

### 🧑 1. User Story
As a [role], I want [goal], So that [benefit].

### 📈 2. Business Impact & Motivation
- **Problem Statement:** What pain point does this solve? Reference current state.
- **Value:** Why is this worthwhile now? Quantify where possible.
- **Measurable Signal:** How will we know this was successful?

### ✅ 3. Acceptance Criteria
Scenario-based Given/When/Then. Must include: happy path, negative/edge case, permission check (if role-based), boundary conditions, non-functional (when relevant).

### 🎨 4. Design & Interaction
- **Design Link:** `[Figma/Design Link: To be added]`
- **User Flow:** Step-by-step navigation path. Reference spec screen descriptions.

### 🛠 5. Functional & Developer Notes
- **Validations:** Field-level rules with error messages
- **Data Requirements:** New/modified fields with constraints
- **Business Rule Changes:** Current → Proposed table (with BR-IDs from spec)
- **Blockers/Dependencies:** Related tickets, systems
- **Affected Spec Sections:** Exact file paths and change descriptions from the change proposals

### 🚫 6. Out of Scope
What this ticket will NOT include. Never empty.

---

## Spec-Updater Specifics

Since the spec-updater always has the existing spec, tickets have richer context:

1. **"What Currently Exists" is always populated from spec** — with file references and IDs
2. **Rule ID continuity** — New rules continue existing numbering (BR-ORD-006 after BR-ORD-005)
3. **User story ID continuity** — Same approach (US-ORD-016 after US-ORD-015)
4. **Cross-module impact** — If change in one module affects another, link the tickets
5. **Change proposal linkage** — Each ticket references its corresponding change proposal file

## Decomposition Rules

### When to create an Epic
- Requirement spans 3+ Stories
- Affects multiple modules
- Introduces a new feature area

### How to decompose
1. **By user action** — each distinct action = one Story
2. **By role** — if different roles have different experiences
3. **By screen** — if adding multiple screens

### When NOT to decompose
- Simple rule change → single Story
- Single field addition → single Story
- Label/text change → single Story

---

## Quality Rules

- Summary always uses action verb format
- "What Currently Exists" always populated from the spec
- User Story "So that" is a real business benefit
- Business Impact has a measurable signal
- At least 1 negative/edge scenario per ticket
- Non-functional AC included when performance or security matters
- Out of Scope is never empty
- No technical jargon in sections 1-4 and 6
- Spec references concentrated in section 5
