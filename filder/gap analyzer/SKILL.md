# Gap Detector — Functional & Technical Gap Analysis

Scan a codebase to find gaps between **what users need** and **what the code delivers**, plus technical defects that could become production bugs.

## When to Use

- Before a production release — "What could go wrong?"
- After generating a functional spec — "Is everything actually implemented?"
- During code audit — "What patterns could cause incidents?"
- Before onboarding a new module — "How solid is this code?"

## Philosophy

Every gap must answer: **"What breaks for the user, and how severe is it?"**

Gaps are not theoretical concerns — they are concrete scenarios where a real user hits a real problem. Rank everything by production impact.

---

## Step 0 — Clarify Scope & Inputs

Before scanning, confirm:

1. **Codebase location(s)** — Path to Java backend, Angular frontend, or both.
2. **Functional Spec** — Does a functional spec already exist?
   - **Yes:** Ask for the file path. This is the requirements baseline.
   - **No:** Suggest running the `functional-spec-generator` skill first to create one, OR proceed with a code-only technical audit.
3. **Target scope** — Whole codebase, specific module, or specific user workflow?
4. **Known pain points** — Ask: "Are there areas you're already worried about?" These get priority scanning.
5. **Output format** — Default is Word document (`.docx`). Can also be Markdown.

If the user says "just scan everything", default to: full scope, both functional + technical, Word output.

## Step 1 — Ingest the Functional Specification (Requirements Baseline)

If a functional spec exists (generated by the `functional-spec-generator` skill or provided by the user):

### 1a. Extract Checkable Requirements

Read the spec and extract a **Requirements Checklist** — a flat list of concrete, verifiable statements:

| Req ID | Module | Requirement | Type |
|--------|--------|-------------|------|
| FR-USR-001 | User Management | A user can reset their password via email | User Story |
| FR-USR-002 | User Management | Email field is required and must be valid format | Data Validation |
| BR-ORD-003 | Orders | Orders above £10,000 require manager approval | Business Rule |
| FR-ORD-004 | Orders | User sees confirmation message after order submission | User Feedback |
| FR-NAV-001 | Navigation | Admin users see "Settings" menu item | Role-Based Access |

Extract from these spec sections:
- **User Stories** → Each story = one requirement
- **Business Rules** → Each rule = one requirement
- **Data Requirements** → Each required field / validation = one requirement
- **Error Handling** → Each documented error scenario = one requirement
- **Screen Descriptions** → Each described interaction (button, form, navigation) = one requirement
- **Cross-Cutting Concerns** → Auth flows, notifications, exports = requirements

### 1b. If No Spec Exists

If no functional spec is available:
- Perform a lightweight version of Step 1 from the `functional-spec-generator` skill (reconnaissance) to build a rough feature inventory
- Use the feature inventory as the baseline, but flag in the output that the analysis is based on **inferred requirements**, not a confirmed spec
- Recommend the user generate a full spec for higher-confidence gap analysis

## Step 2 — Functional Gap Scan (Spec vs. Code)

For each requirement in the checklist, search the code for its implementation. Classify each as:

### Gap Categories

| Status | Meaning | Icon |
|--------|---------|------|
| ✅ Implemented | Code fully satisfies the requirement | — |
| ⚠️ Partial | Code exists but is incomplete or handles only the happy path | Warning |
| ❌ Missing | No code found that implements this requirement | Critical |
| 🔍 Ambiguous | Code exists but intent is unclear; may or may not satisfy the requirement | Needs review |

### How to Verify Each Requirement Type

**User Stories ("As a [role], I can [action]")**
```bash
# Find the endpoint or UI component that handles the action
grep -rn "@PostMapping\|@GetMapping\|@PutMapping\|@DeleteMapping" <java_path> --include="*.java" | grep -i "<action_keyword>"
# Find the Angular component/route
grep -rn "path:\|component:" <angular_path> --include="*.ts" | grep -i "<feature_keyword>"
```
- ✅ if endpoint + service logic + UI component all exist
- ⚠️ if endpoint exists but no UI, or UI exists but no backend, or only happy path handled
- ❌ if no matching code found

**Business Rules ("When X, then Y")**
```bash
# Search for conditional logic related to the rule
grep -rn "if\|switch\|when\|threshold\|limit\|max\|min\|approve\|reject" <java_path>/service/ --include="*.java" | grep -i "<rule_keyword>"
```
- ✅ if the exact condition and outcome are coded
- ⚠️ if the condition is checked but the outcome is wrong/incomplete (e.g. no error message to user)
- ❌ if no conditional logic matches the rule

**Data Validations ("Field X is required, max 100 chars")**
```bash
# Check entity annotations
grep -rn "@NotNull\|@NotBlank\|@Size\|@Pattern\|@Email\|@Min\|@Max\|@Valid" <java_path> --include="*.java" | grep -i "<field_keyword>"
# Check Angular form validators
grep -rn "Validators\.\|required\|minLength\|maxLength\|pattern" <angular_path> --include="*.ts" | grep -i "<field_keyword>"
```
- ✅ if both backend and frontend validate
- ⚠️ if only one side validates (frontend-only validation is a gap — users can bypass it)
- ❌ if no validation exists

**Error Handling ("When X fails, user sees message Y")**
```bash
# Find exception handling for the scenario
grep -rn "catch\|@ExceptionHandler\|ControllerAdvice\|throw" <java_path> --include="*.java" | grep -i "<error_keyword>"
# Find UI error display
grep -rn "error\|toast\|snackbar\|alert\|message" <angular_path> --include="*.html" --include="*.ts" | grep -i "<error_keyword>"
```
- ✅ if exception is caught, meaningful message returned, and UI displays it
- ⚠️ if exception is caught but generic "Something went wrong" is shown
- ❌ if exception is not caught (user sees raw stack trace or white screen)

**Role-Based Access ("Only [role] can access [feature]")**
```bash
grep -rn "@PreAuthorize\|@Secured\|@RolesAllowed\|hasRole\|hasAuthority" <java_path> --include="*.java" | grep -i "<feature_keyword>"
grep -rn "canActivate\|AuthGuard\|role" <angular_path> --include="*.ts" | grep -i "<feature_keyword>"
```
- ✅ if both backend and frontend enforce the role check
- ⚠️ if only frontend guards exist (backend is unprotected — security gap)
- ❌ if no role check exists

## Step 3 — Technical Gap Scan (Code-Level Defects)

Independently of the functional spec, scan the code for patterns known to cause production incidents. These are organized by **severity tier**.

### Tier 1 — Critical (Will likely cause production incidents)

#### 1.1 Silent Exception Swallowing
```bash
# Find empty catch blocks or catch-and-ignore patterns
grep -rn -A2 "catch\s*(" <java_path> --include="*.java" | grep -B1 "// \|{$\|log.debug\|//TODO\|// todo"
# Better: find catch blocks with no logging at WARN/ERROR level
grep -rn -A5 "catch\s*(" <java_path> --include="*.java" | grep -v "log.error\|log.warn\|LOG.error\|LOG.warn\|throw\|rethrow"
```
**Why it matters:** Errors happen silently, data gets corrupted, users see stale/wrong data with no error message.

#### 1.2 Missing Null Checks on External Inputs
```bash
# Find service methods that receive objects but don't validate
grep -rn "public.*\(.*Request\|.*DTO\|.*Command" <java_path>/service/ --include="*.java"
# Check if those methods start with validation
```
**Why it matters:** NullPointerException in production → 500 error → user sees white screen.

#### 1.3 Unclosed Resources
```bash
# Find resource usage without try-with-resources
grep -rn "new FileInputStream\|new BufferedReader\|new Connection\|getConnection()\|openStream()\|new Socket" <java_path> --include="*.java"
# Check if these are inside try-with-resources or have finally blocks
```
**Why it matters:** Resource leaks → connection pool exhaustion → system-wide outage.

#### 1.4 Missing Timeouts
```bash
# Find HTTP clients, RestTemplate, WebClient without timeout config
grep -rn "RestTemplate\|WebClient\|HttpClient\|OkHttpClient\|Feign" <java_path> --include="*.java"
# Find database query execution without timeouts
grep -rn "createQuery\|createNativeQuery\|jdbcTemplate" <java_path> --include="*.java"
```
**Why it matters:** One slow downstream service → thread pool starvation → cascading failure.

#### 1.5 Unbounded Collections
```bash
# Find queries without LIMIT or pagination
grep -rn "findAll()\|SELECT.*FROM.*WHERE" <java_path> --include="*.java" | grep -v "Pageable\|LIMIT\|setMaxResults\|Top\|First"
```
**Why it matters:** Table grows → one query returns 1M rows → OutOfMemoryError → crash.

### Tier 2 — High (Will cause bugs under load or edge cases)

#### 2.1 Race Conditions
```bash
# Find shared mutable state
grep -rn "static.*Map\|static.*List\|static.*Set\|static.*=\s*new\|@Singleton.*Map\|@Component.*Map" <java_path> --include="*.java" | grep -v "final.*Collections.unmodifiable\|final.*Map.of\|ConcurrentHashMap"
# Find non-atomic read-modify-write
grep -rn "\.get(.*\.put(\|\.get(.*\.set(\|\.size().*\.add(" <java_path> --include="*.java"
```

#### 2.2 Missing @Transactional or Wrong Propagation
```bash
# Find service methods that do multiple writes without @Transactional
grep -rn "\.save(\|\.delete(\|\.update(" <java_path>/service/ --include="*.java"
# Cross-reference with @Transactional annotations
grep -rn "@Transactional" <java_path>/service/ --include="*.java"
```
**Why it matters:** Partial writes → data inconsistency → user sees corrupt data.

#### 2.3 Frontend-Only Validation
```bash
# List all Angular validators
grep -rn "Validators\." <angular_path> --include="*.ts" | sed 's/.*Validators\.//' | sort -u
# Compare against backend validation annotations on same fields
```
**Why it matters:** Any API caller (or browser devtools) bypasses frontend checks → bad data enters system.

#### 2.4 Missing Error Responses to UI
```bash
# Find controllers that return void or don't wrap errors
grep -rn "public void\|ResponseEntity.ok()\|return null" <java_path> --include="*.java" | grep -i "controller"
# Find catch blocks that return generic 500
grep -rn -A3 "catch" <java_path> --include="*.java" | grep "INTERNAL_SERVER_ERROR\|HttpStatus.500\|status(500)"
```

#### 2.5 Hardcoded Values
```bash
# Find hardcoded URLs, IPs, ports, credentials
grep -rn "http://\|https://\|jdbc:\|password\s*=\|secret\s*=\|apiKey\s*=" <java_path> --include="*.java" --include="*.properties" --include="*.yml" | grep -v "test\|Test\|example.com\|localhost"
```

### Tier 3 — Medium (Will cause subtle bugs or maintenance issues)

#### 3.1 Inconsistent equals/hashCode
```bash
# Find classes with equals but no hashCode or vice versa
grep -rln "equals(" <java_path> --include="*.java" | while read f; do
  has_hash=$(grep -c "hashCode" "$f")
  if [ "$has_hash" -eq 0 ]; then echo "MISSING hashCode: $f"; fi
done
```

#### 3.2 Logging Sensitive Data
```bash
grep -rn "log\.\|LOG\.\|logger\." <java_path> --include="*.java" | grep -i "password\|ssn\|credit.card\|token\|secret\|api.key\|authorization"
```

#### 3.3 Missing Enum Default/Exhaustive Handling
```bash
# Find switch statements on enums without default
grep -rn -A20 "switch\s*(" <java_path> --include="*.java" | grep -B20 "case" | grep -v "default"
```

#### 3.4 Thread Pool Misconfigurations
```bash
grep -rn "Executors.newFixedThreadPool\|newCachedThreadPool\|newSingleThreadExecutor\|ThreadPoolExecutor\|@Async" <java_path> --include="*.java"
# Check for rejection policies and queue bounds
```

#### 3.5 Missing Index Hints for Slow Queries
```bash
# Find @Query with complex WHERE clauses
grep -rn "@Query" <java_path> --include="*.java" | grep -i "WHERE.*AND.*AND"
# Find native queries with JOINs
grep -rn "nativeQuery\s*=\s*true" <java_path> --include="*.java"
```

## Step 4 — Workflow Integrity Check

Beyond individual gaps, check complete user workflows end-to-end:

### 4.1 Identify Key Workflows
From the functional spec (or inferred from code), list the critical user journeys:
- E.g., "New customer onboarding: Register → Verify email → Complete profile → First order"

### 4.2 Trace Each Workflow Through Code
For each step in the workflow:
- Does the code handle the transition between steps?
- What happens if the user abandons mid-workflow? (orphaned records? locked resources?)
- What happens if the user retries a step? (duplicate records? idempotency?)
- What happens if concurrent users execute the same workflow on the same entity?

### 4.3 Common Workflow Gaps

| Gap Pattern | How to Detect | Production Impact |
|-------------|--------------|-------------------|
| Missing idempotency | `POST` endpoints without duplicate-check logic | Double-charges, duplicate records |
| No state validation | Status transitions without checking current state | Order shipped before payment confirmed |
| Orphaned child records | Parent delete without cascade or child cleanup | Data ghosts, broken references, FK violations |
| Missing async error handling | `@Async` methods with no error callback | Silent failures in background jobs |
| Stale cache | Cache without TTL or invalidation on write | Users see outdated data |
| Missing pagination | List endpoints returning full result sets | Works in dev, OOM in prod with real data |

## Step 5 — Assemble the Gap Analysis Report

Structure the output document as follows:

### Report Structure

1. **Executive Summary**
   - Total gaps found by severity (Critical / High / Medium)
   - Top 5 risks ranked by production impact
   - One-paragraph recommendation

2. **Functional Gap Analysis** (if spec was provided)
   - Requirements coverage summary: X of Y requirements fully implemented, Z partial, W missing
   - Gap detail table per module:

     | Req ID | Requirement | Status | Gap Description | Impact | Recommended Fix |
     |--------|-------------|--------|-----------------|--------|-----------------|
     | FR-USR-001 | Password reset via email | ⚠️ Partial | Email is sent but no expiry on reset token | User can reuse old reset links indefinitely | Add 24-hour token expiry |

3. **Technical Gap Analysis**
   - Findings grouped by Tier (Critical → High → Medium)
   - For each finding:

     | ID | Category | File/Location | Description | Production Impact | Recommended Fix | Effort |
     |----|----------|--------------|-------------|-------------------|-----------------|--------|
     | TG-001 | Silent Exception | OrderService.java:142 | Catch block logs at DEBUG, no rethrow | Failed orders silently lost | Log at ERROR, add alerting, rethrow | Low |

4. **Workflow Integrity**
   - For each audited workflow: status (Clean / Gaps Found) and gap details

5. **Risk Heatmap Summary**
   - Matrix: Module × Gap Type, color-coded by severity
   - Helps prioritize which modules to fix first

6. **Recommendations & Prioritized Fix Plan**
   - Ordered list: fix Critical first, then High, then Medium
   - Group by module for efficient developer assignment
   - Estimate effort: Low (< 1 day), Medium (1-3 days), High (3+ days)

## Quality Checklist — Apply Before Delivering

- [ ] **Every gap has a concrete scenario** — Not "might fail" but "When user X does Y, Z happens because..."
- [ ] **Every gap has production impact** — What does the user experience? Data loss? Wrong results? Crash?
- [ ] **No false positives from test code** — Exclude `src/test/`, `*Spec.ts`, `*.spec.ts`, `*Test.java`
- [ ] **Functional gaps reference the spec** — Each gap links back to a Req ID
- [ ] **Technical gaps include file and line** — Developer can find the problem immediately
- [ ] **Recommendations are actionable** — Not "fix the bug" but "Add @NotNull annotation to customerId parameter in OrderService.createOrder()"
- [ ] **Severity is justified** — Critical = data loss or outage. High = bug under load. Medium = edge case.
- [ ] **Effort estimates are realistic** — Consider testing effort, not just code change

---

## Working with Large Codebases

1. Start with Step 1 (requirements extraction) for the full spec
2. Run Step 3 (technical scan) across the full codebase — the grep patterns scale well
3. For Step 2 (functional gap scan), work module by module
4. Present findings per module and ask: "Should I continue to the next module or dig deeper here?"
5. Compile final report after all modules are scanned
