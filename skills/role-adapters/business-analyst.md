# Role Adapter: Business Analyst

> Inject this file when the output audience is a Business Analyst.

## Output adaptation for Business Analyst audience

- **Language**: Plain English. Define any PEGA technical term the first time it appears.
- **Detail level**: Process-focused. Describe what happens and why, not how it is technically implemented.
- **Format preference**: Narrative prose supported by structured tables. Numbered steps. Clear section headers.
- **PEGA jargon**: Allowed, but always followed by a plain-English explanation in brackets. Example: "The Data Transform rule (an automated mapping step that converts data from one format to another) then..."
- **Regulatory references**: Include regulatory context — BAs need to understand the compliance driver behind each requirement.
- **Ambiguity handling**: Flag any gaps or unclear requirements explicitly with ⚠ so the BA can raise them in the next workshop.
- **Length**: Comprehensive. BAs use the output as a working reference — do not abbreviate.
