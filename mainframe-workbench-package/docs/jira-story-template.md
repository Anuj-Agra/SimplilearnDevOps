# Jira Story Format — Mainframe Renovation Reference

Use this as a guide when importing stories generated by the Jira Stories agent into your Jira project.

---

## Epic Template

```
Epic Key:    EP-[N]
Epic Name:   [Business-friendly name — e.g. "Order Management Migration"]
Description: Migrate the [CICS-TXN-ID] transaction family from Natural/COBOL to
             Java Spring Boot API + Angular frontend. Covers [N] programs, [N] Adabas
             files, and [N] reference tables.
Label:       mainframe-migration, [functional-area]
Target:      Sprint [N] — [date]
```

## Story Template

```
Story Key:   US-[N]
Epic Link:   EP-[N]
Summary:     As a [role], I want [feature] so that [benefit]
Description:
  Migrates functionality from Natural program [PROGRAM-NAME] / CICS transaction [TXN-ID].

  Original behaviour:
  - [description of what the mainframe program does]

  Renovation scope:
  - Angular: [ComponentName] in module [ModuleName]
  - API:     [HTTP verb] /api/[path]
  - Service: [ServiceClassName].method()
  - Entity:  [EntityName] (from Adabas FNR=[n])

Story Points: [1 / 2 / 3 / 5 / 8 / 13]
Priority:     [Must Have / Should Have / Could Have / Won't Have]
Labels:       renovation, [frontend|backend|data-migration]
```

## Acceptance Criteria (Gherkin Format)

Every story must have at minimum one happy path scenario and one validation/error scenario.

### Happy path
```gherkin
Given the user has [precondition — e.g. logged in with role X]
  And [data precondition — e.g. customer record exists with status Active]
When the user [action — e.g. submits the order form with valid data]
Then the system [expected outcome — e.g. creates an order record]
  And the user sees [UI feedback — e.g. confirmation message with order number]
  And the API returns [HTTP 201 with order ID in response body]
```

### Validation scenario
```gherkin
Given the user is on the [screen name] screen
When the user submits the form with [invalid condition — e.g. blank mandatory field]
Then the form displays error "[error message from original Natural validation]"
  And no record is created in the database
  And the API returns HTTP 400 with field-level error details
```

### Business rule scenario
```gherkin
Given a customer with credit limit [£X]
  And existing outstanding orders totalling [£Y]
When a new order is submitted for [£Z] where Y + Z > X
Then the system rejects the order with message "[BR-012: Credit limit exceeded]"
  And the rejection is logged with customer ID and attempted amount
```

## Technical Sub-task Template

For each story, create the following sub-tasks:

```
[US-001] FE: Create Angular [FeatureName]Component
  → File: src/app/[module]/[feature]/[feature].component.ts
  → Inputs from parent route / resolver
  → Reactive form with validators mirroring BR-[n]

[US-001] FE: Create [FeatureName]Service (API client)
  → File: src/app/[module]/[feature]/[feature].service.ts
  → Calls: [HTTP verb] /api/[path]

[US-001] BE: Create [FeatureName]Controller
  → File: src/main/java/.../controller/[Feature]Controller.java
  → Endpoints: [list]

[US-001] BE: Create [FeatureName]Service
  → File: src/main/java/.../service/[Feature]Service.java
  → Business logic from Natural program [PROGRAM-NAME]

[US-001] BE: Create [EntityName]Entity + [EntityName]Repository
  → File: src/main/java/.../entity/[Entity].java
  → Mapped from Adabas FNR=[n], DDM fields: [list]

[US-001] TEST: Write unit tests for [FeatureName]Service
  → Cover: happy path, validation failures, business rule edge cases
  → Baseline against mainframe behaviour

[US-001] TEST: Write Angular component spec
  → Cover: form validation, API call on submit, error display
```

## Renovation-Specific Epic Templates

### Data Migration Epic
```
EP-DM: Data Migration — Adabas to [Target DB]

Stories:
  US-DM-001: Extract Adabas FNR=[n] ([FileName]) to staging CSV
  US-DM-002: Validate staging data against business rules
  US-DM-003: Transform and load into [target table]
  US-DM-004: Reconcile record counts and key field checksums
  US-DM-005: Handle MU/PE field denormalisation for [FieldName]
```

### Reference Table Migration Epic
```
EP-RT: Reference Table Migration

Stories:
  US-RT-001: Catalogue all reference tables and entry counts
  US-RT-002: Migrate [TABLE-NAME] to Spring @Configuration enum
  US-RT-003: Migrate [TABLE-NAME] to database lookup table with cache
  US-RT-004: Decommission Natural reference table load programs
```

### Regression Testing Epic
```
EP-REG: Regression Testing

Stories:
  US-REG-001: Capture mainframe golden data set for [TXN-ID]
  US-REG-002: Run parallel comparison — mainframe vs Java API for [scenario]
  US-REG-003: Automate regression suite in CI/CD pipeline
  US-REG-004: Sign-off regression report with business owner
```

## Priority Classification Guide

| MoSCoW | Criteria for mainframe migration |
|--------|----------------------------------|
| **Must Have** | Core transaction functionality; legally required; data loss risk if absent |
| **Should Have** | Important feature; workaround exists but is unacceptable long-term |
| **Could Have** | Nice to have; present in mainframe but rarely used; low business risk if deferred |
| **Won't Have** | Identified as obsolete by Obsolescence agent; no active users; legacy technical debt |
